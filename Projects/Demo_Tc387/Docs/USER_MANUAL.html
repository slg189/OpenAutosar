1 · 这是什么

OpenAutosar 是一个 AUTOSAR Classic Platform 嵌入式工程框架，面向 Infineon AURIX TC3xx（TC387/TC367）， 覆盖 MCAL、复杂驱动 (CDD)、BSW、组件 (Components)、上层应用 (ASW) 五层，并提供一套 v3 构建框架： 用户主要改一份 Projects/<项目>/build.yaml 即可完成 编译 / 代码生成 / 文档 / 静态检查 / 单元测试 / 远程构建。

一句话上手： python Tools/setup_env.py（搭环境）→ 激活 venv → scons compile TOOLCHAIN=gcc PLATFORM=host（无需 TriCore 工具链先验证）。

2 · 核心概念（先懂这几个词）

概念含义
层 (layer)五层：MCAL / CDD / ASW / BSW / Components
模块 (module)层下的一个功能单元 = 一个目录（含 inc/ 头文件，可含 src/ 源码）。如 MCAL/Tc387/Adc、BSW/Com
mode: source / lib该模块从源码编译（source）还是链接预编译库（lib，取自 <层>_Libs/ 的 .a）
visibility: auto未显式写 mode 时，框架按“工作区是否真有该模块源码”自动选 source/lib（信息安全分级用）
工具链 / 平台TOOLCHAIN（hightec / tasking / gcc…）+ PLATFORM（AURIX2G / host…），可插拔适配器
*_Libs/各层的二进制释放区（ASW_Libs/CDD_Libs/BSW_Libs/MCAL_Libs），存 .a+头文件，供 lib 模式链接 / 信息安全下发
Integration工程下的集成/胶水层（main.c、RTE 连接、EcuM 回调），由基础工程师维护

4 · PC 环境搭建与检查

目标：各机器用同一份固定版本的工具，避免“在我电脑上能跑”。详见 Tools/ENVIRONMENT.md。

4.1 一键搭建

# 1) 一条命令: 校验外部工具 + 下载固定工具(cmake/googletest) + 建隔离 venv + 装 pinned 依赖
python Tools/setup_env.py

# 2) 激活隔离环境 (与系统 Python/SCons 隔离)
source Tools/.venv/bin/activate      # Windows: Tools\.venv\Scripts\activate
#   或不激活直接用:  Tools/.venv/bin/scons

# 3) 验证 (无需 TriCore 工具链, 用系统 gcc 真编一遍)
scons compile TOOLCHAIN=gcc PLATFORM=host

4.2 只检查环境（不安装）

python Tools/setup_env.py --check     # 列出必需/可选工具是否齐备 + cmake>=3.16 版本硬校验
python Tools/setup_env.py --no-fetch  # 只建 venv, 不下载工具

5.1 信息安全分级（source/lib 自动决策）

不同角色用不同 manifest 签出：能看到源码的层 → 编译；看不到的 → 自动链接 *_Libs/ 的 .a。 在 build.yaml 里 defaults.visibility: auto 即开启此自动机；也可对单模块写死 mode: source|lib。

5.2 灵活编译

每模块可独立配 mode、enabled、cflags、extra_defines、extra_includes；命令行可临时过滤 only= / skip= / layers=，并用 TOOLCHAIN=/PLATFORM=/BUILD_TYPE=/STRICT=1 覆盖。

6 · build.yaml 配置详解（用户主要改这份）

schema_version: "3.0"
project: { name: Demo, derivative: Tc387 }      # 产物名 = Demo_Tc387(.elf)
target:
  toolchain: hightec        # 目标工具链 (可命令行 TOOLCHAIN= 覆盖)
  platform:  AURIX2G        # 平台
  build_type: Release       # Debug | Release
defaults:
  visibility: auto          # auto: 按工作区可见性自动 source/lib
  chip: Tc387               # MCAL 等按芯片的层

global_defines:  [AUTOSAR_CP, PROJECT_Demo_Tc387]
global_includes: ["${MCAL_DIR}/inc", "${COMP_DIR}/Common/inc", "${PROJECT_DIR}/Gen"]

layers:                     # ★ 显式列要构建的模块 (每层一段)
  MCAL:
    Adc:  { mode: source, cflags: [-O3] }   # 源码编译, 加模块级 flag
    Port: { mode: source }
    Can:  { mode: lib }                     # 链接 MCAL_Libs/Can/libCan.a
  CDD:
    CddPwm:                                 # release 时额外发布 gen 配置头
      mode: source
      release_includes:
        - CDD/CddPwm/gen/CddPwm_Cfg.h      # 显式列出 → CDD_Libs/CddPwm/inc/
    CddSent:  { mode: lib }
  ASW:
    VehicleControl:   { mode: source }
    EnergyManagement: { mode: lib }
  BSW:
    Os:  { mode: source, lib_name: libRtaOs }   # 释放时 .a 命名为 libRtaOs.a
    Com: { mode: source }
    Crc: { mode: source }
    # CanIf:
    #   mode: source
    #   release_includes:                   # 工程 Gen 里的配置头
    #     - ${PROJECT_DIR}/Gen/BSW/CanIf/CanIf_Cfg.h
  Components:
    Common:  { mode: source }
    Mcu/Sbm: { mode: source }               # 嵌套路径模块

integration: { enabled: true, dir: Integration }
linker:  { script: "", libraries: [], extra_flags: [] }   # 有链接脚本填 Ld/Demo_Tc387.ld
output:  { formats: [elf, hex, map], size_report: true, compile_commands_json: true }
codegen: { enabled: true, generators: [ { name: demo_echo, type: custom, cmd: 'echo gen' } ] }
docs:    { enabled: true, output_dir: "${PROJECT_DIR}/Docs/api" }
remote:  { enabled: false }
模块字段作用
mode: source|lib从源码编译 / 链接预编译库；省略则按 defaults.visibility
enabled: false本次不构建该模块
cflags: [..]模块级编译选项（追加）
extra_defines / extra_includes模块级宏 / 头文件路径
lib_name: libXxx覆盖 release/链接时的 .a 基名（默认 lib<模块>）
release_includes: [文件, ...]release 时需额外发布到 *_Libs/<模块>/inc/ 的具体头文件路径（相对仓库根或 ${VAR}）。
静态头 <模块>/inc/*.h 始终自动收集；配置/生成头需在此显式列出，如：
- CDD/CddPwm/gen/CddPwm_Cfg.h
- ${PROJECT_DIR}/Gen/BSW/CanIf/CanIf_Cfg.h

7 · 模块的添加与删除

7.1 添加一个模块（源码模块）

两步：建目录骨架 → 在 build.yaml 声明。（光建目录只会被 scons list “发现并提示”，声明后才会构建。）

# ① 生成目录骨架 (inc/<名>.h + src/<名>.c 桩)
python Tools/scripts/new_module.py --bsw CanIf          # BSW/CanIf/
python Tools/scripts/new_module.py --mcal Tc387/Eth     # MCAL 带芯片
python Tools/scripts/new_module.py --comp Mcu/Wdg       # Components 嵌套
#   (也可手动建目录, 放好 inc/ 与 src/)

# ② 在 Projects/Demo_Tc387/build.yaml 的 layers 下加一行:
#    BSW:
#      CanIf: { mode: source, extra_includes: ["${BSW_DIR}/Can/inc"] }

# ③ 确认并构建
scons list                                  # 应能看到 CanIf 在计划里
scons compile TOOLCHAIN=gcc PLATFORM=host   # 或 scons (目标工具链)

7.2 添加一个“预编译库”模块（lib 模式）

把交付的 .a + 头文件放到对应释放区，再在 build.yaml 写 mode: lib：

# 例: 加一个只给 .a 的 CDD 模块 CddLin
#   CDD_Libs/CddLin/libCddLin.a   +   CDD_Libs/CddLin/inc/CddLin.h
# build.yaml:
#   CDD:
#     CddLin: { mode: lib }
scons explain:CddLin        # 确认解析到 CDD_Libs/CddLin/libCddLin.a

7.3 删除 / 停用一个模块

目的做法
临时不编（命令行）scons skip=CddMotor（或只编子集 only=/layers=）
本工程长期停用build.yaml 里该模块改 { enabled: false }，或删除该模块条目
彻底移除删除 build.yaml 条目 且 删除模块目录（git rm -r <层>/<模块>）；如有释放库再删 <层>_Libs/<模块>
注意：只删 build.yaml 条目而留着目录 → 该模块仍在 scons list 的“未声明”提示里，但不会被构建。若删了目录却忘了删 build.yaml 条目 → mode: source 会因找不到源码而明确报错（fail-fast）。

7.4 source ↔ lib 互换

改 build.yaml 里该模块的 mode 即可：source（编译本层源码）↔ lib（链接 <层>_Libs 的 .a）。 用 scons release 可把当前源码模块编成 .a 回灌到 *_Libs/，供下游以 lib 集成。

7.5 release 头文件发布规则

scons release 会自动把模块 inc/*.h（静态 API 头）发布到 *_Libs/<模块>/inc/。 但配置/生成头（如 CddPwm_Cfg.h、CanIf_Cfg.h）不会自动收集， 需在 build.yaml 的 release_includes 中逐文件显式列出：

CDD:
  CddPwm:
    mode: source
    release_includes:
      - CDD/CddPwm/gen/CddPwm_Cfg.h               # 模块级 gen 配置头
BSW:
  CanIf:
    mode: source
    release_includes:
      - ${PROJECT_DIR}/Gen/BSW/CanIf/CanIf_Cfg.h    # 工程级 Gen 配置头
      - ${PROJECT_DIR}/Gen/BSW/CanIf/CanIf_PBcfg.h  # 支持 ${VAR} 展开
为什么不自动扫描 gen/ 目录？——Gen 目录可能包含仅编译期需要的内部头文件或中间产物， 不应全部对外发布。显式列出可精确控制下游看到的接口，避免信息泄露。

8 · 命令大全（作用 + 用法）

除特别说明，均在仓库根、激活 venv 后运行。

8.1 scons 目标（targets）

命令作用
scons构建默认项目（编译+链接 ELF；默认 hightec/AURIX2G，需 TriCore 工具链）
scons compile只编译源码模块为 .o，不链接（常配 TOOLCHAIN=gcc PLATFORM=host 做验证）
scons list打印构建计划（模块 / mode / source或*_Libs 来源）+ 未声明模块提示
scons explain:<模块>解释某模块为何 source/lib、include/lib 路径（如 scons explain:Adc）
scons gen / scons gen:<名>运行（指定）代码生成器
scons doc / doc-clean生成 / 清理 API 文档
scons release源码模块 → .a(Libs/<层>/) + inc/*.h + release_includes 显式列出的 gen/cfg 头 → 各 *_Libs/
scons checkcppcheck 真实bug门禁 + MISRA(资讯→Reports/misra.xml) + GoogleTest 单测

8.2 命令行变量（VARIABLE=value）与过滤

变量作用示例
PROJECT=选工程（默认 Projects/Demo_Tc387）scons PROJECT=Projects/Demo_Tc387
TOOLCHAIN= / PLATFORM=覆盖目标工具链/平台scons TOOLCHAIN=gcc PLATFORM=host
BUILD_TYPE=Debug / Releasescons BUILD_TYPE=Debug
STRICT=1host gcc 加 -Werror（告警即错）scons compile TOOLCHAIN=gcc PLATFORM=host STRICT=1
only=A,B仅构建这些模块（名或 leaf）scons only=Adc,CddPwm
skip=X跳过这些模块scons skip=EnergyManagement
layers=MCAL,ASW仅构建这些层scons layers=MCAL
CONFIG=换配置文件名（默认 build.yaml）scons CONFIG=build.debug.yaml
--remote= / --no-pull / --skip-gen远程构建 / 不回传 / 跳过代码生成scons --remote=auto

8.3 脚本命令（python Tools/...）

9 · 典型工作流

9.1 新机器首次上手

git clone <repo> && cd OpenAutosar
python Tools/setup_env.py                      # 环境 + 工具 + venv
source Tools/.venv/bin/activate
scons list                                     # 看构建计划
scons compile TOOLCHAIN=gcc PLATFORM=host      # host 验证 (绿表示源码/框架 OK)
scons check                                    # 静态检查 + 单测

9.2 改某模块并验证

scons explain:Adc                              # 确认它怎么编
scons compile only=Adc TOOLCHAIN=gcc PLATFORM=host   # 只编它, 快速反馈
scons check                                    # 跑 cppcheck + 单测

9.3 出目标固件（有 TriCore 工具链）

call Tools\Compiler\setup_env.bat              # 设置 tricore-gcc 环境
scons -j8                                      # -> Projects/Demo_Tc387/Out/Demo_Tc387.elf (+hex/map)
python Tools/scripts/run_integration_test.py   # QEMU 集成测试 (需 qemu-system-tricore)

9.4 给下游发布二进制

# 确保 build.yaml 各模块的 release_includes 已列好需要发布的 gen/cfg 头文件
scons release                                  # 源码模块 -> Libs/<层>/*.a + *_Libs/<模块>/(*.a + inc/*.h)
# 下游把对应层在 build.yaml 改 mode: lib 即可只拿二进制

10 · CI 与两道质量闸

仓库自带 GitHub Actions（.github/workflows/）：

工作流触发内容
build-validation.yml（第一道闸）push / PRdiscover 发现所有 Projects/*/build.yaml → 每项目：scons list + host gcc -Werror 编译 + 目标构建图 dry-run + scons check（cppcheck/MISRA + 单测）
integration-qemu.yml（第二道闸）手动（workflow_dispatch）在自托管 runner（标签 tricore）上构建 ELF + QEMU 集成测试；不阻塞 PR
第一道闸不需要 TriCore 工具链（host gcc），PR 必过；第二道闸需你注册带 tricore-gcc+qemu-system-tricore 的自托管 runner 后手动触发。

11 · 故障排查 / FAQ

现象原因 / 解决
scons 报找不到 tricore-gcc默认目标是 AURIX。验证用 scons compile TOOLCHAIN=gcc PLATFORM=host；出固件需装 HighTec 并 setup_env.bat
[FATAL] ... 源码不可见且未找到释放库该模块 mode: source 但没源码，或 mode: lib 但 *_Libs 没有 .a。scons explain:<模块> 看它找了哪些路径
新建模块没被编译只建目录不够，要在 build.yaml 的 layers 下声明（见 §7.1）；scons list 会把它列在“未声明”里提示
scons --explain Adc 报 no such option正确写法是目标形式 scons explain:Adc
单元测试拉不到 googletest（离线）跑 python Tools/fetch_tools.py 把 googletest 下到 .toolcache，run_gtest 会自动用本地源码
cppcheck 未安装scons check 会跳过静态检查只跑单测；装：apt install cppcheck
系统 Python/SCons 版本不对用隔离 venv：python Tools/setup_env.py 后 source Tools/.venv/bin/activate
QEMU 集成测试被跳过缺 qemu-system-tricore 或 ELF（无 TriCore 工具链）→ 记 skipped，属正常；在自托管 runner 上跑
