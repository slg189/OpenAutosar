1 · 目的

本仓目标芯片为 Infineon AURIX TC3xx，正式构建用 hightec(tricore-gcc) / tasking 工具链。 但在没有 TriCore 工具链的开发/CI 环境里，仍希望真实验证：源码树是否能编、框架的 include 连通性是否正确。

由于 v3 构建框架工具链/平台可插拔，我们新增 toolchain=gcc + platform=host 两个适配器， 用系统 gcc 真实编译各源码模块为 .o（scons compile 目标，只编译不链接）。 这比 scons -n（仅构建图）强得多——它真的调用编译器。

关键收获：本次 host-gcc 验证抓到了 2 个 scons -n 永远发现不了的真实 bug（见 §5）。 “能画出构建图”≠“真能编过”。

2 · 怎么跑

# 仓库根 (需有系统 gcc; 不需要 TriCore 工具链)
scons compile TOOLCHAIN=gcc PLATFORM=host

# 也可只验证某层/某模块
scons compile TOOLCHAIN=gcc PLATFORM=host layers=MCAL
scons compile TOOLCHAIN=gcc PLATFORM=host only=Crc,Adc

compile 是 v3 新增的别名：只把源码模块编译成 .o（含集成层），不链接。 这样可绕开预编译 .a 是 TriCore 目标码、无法在主机链接的问题（见 §6）。

适配器文件作用
toolchain=gccsite_scons/autosar/toolchain/gcc.py系统 gcc，通用 C 标志（无嵌入式 flag）
platform=hostsite_scons/autosar/platform/host.py本机架构，不注入目标 CPU flag

5 · 抓到的真实 bug（dry-run 漏掉）

#问题根因修复
B1 所有 Python 动作（codegen / docgen / check）真执行即崩：
TypeError: _run_codegen() got an unexpected keyword argument 'env'		 SCons 以关键字 env= 调用函数动作，而函数把第三参命名为 env_。-n 不执行动作，故从未暴露。 动作签名统一改为 (target, source, env)（base.py / docgen / check）。
B2 ASW/VehicleControl 编译失败：
fatal error: Rte_Type.h: No such file or directory		 v3 未把各层公共头 <Layer>/inc（如 ASW/inc/Rte_Type.h）加入 include 路径；旧 SConscript 曾手工加 #ASW/inc。 编排器全局补 <Layer>/inc（ASW/CDD/MCAL/BSW/Components 存在即加）。
这正是“真编一次”的价值：两个问题在 scons -n 下都是绿的。

6 · 局限

  • 只编译、不链接：预编译 .a（Can/CddSent/EnergyManagement）是 TriCore 目标码，无法在主机链接成可执行；故验证止于 .o。
  • 非目标 ABI：host gcc 用本机 ABI/类型宽度，能查语法/头文件/宏，但查不出目标相关问题（对齐、位宽、寄存器、链接脚本）。真机/QEMU 验证仍需 TriCore 工具链。
  • 对仅以 .a 交付、无源码的模块（如 EnergyManagement）天然无法做源码级验证。

定位：host-gcc 验证是廉价的第一道闸（语法/include/框架连通性），不替代目标工具链的编译链接与 QEMU 集成测试。

7 · 后续 / CI

建议把 scons compile TOOLCHAIN=gcc PLATFORM=host 作为 CI 的快速门禁（PR 必过）。详细推进见 环境验证开发计划 (MD)。要点：

  • CI job：装 scons+pyyaml+gcc → scons compile TOOLCHAIN=gcc PLATFORM=host。
  • 逐步开 -Werror（先 demo 桩代码达标）。
  • 多项目/多 build.yaml 矩阵；scons list 做计划快照。
  • 有 TriCore 环境的 runner 上跑真机编译/链接 + QEMU 集成测试（第二道闸）。
